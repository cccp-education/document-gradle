== Vision Produit

Un plugin Gradle N2 qui **cree du contenu AsciiDoc assiste par IA**
et le **publie dans tous les formats supportes par AsciidoctorJ**
(HTML5, PDF, EPUB3, DocBook 5, ManPage), sans sortir du pipeline Gradle.

Le plugin couvre deux missions complementaires :

1. **Generation IA** → un LLM (deepseek-v4-pro) assiste la redaction
   de documentation, specs, manuels, rapports en AsciiDoc structure
2. **Publication multi-format** → chaque backend AsciidoctorJ est expose
   comme une tache Gradle isolee avec configuration fine

'''
== Boundary Codex (Brooklyn)

_Document = WRITE + PUBLISH. Codex = READ + RAG._

Voir `.agents/ARCHITECTURE_BOUNDARY.adoc` pour le detail complet.

| Sortie | HTML, PDF, EPUB, DocBook, ManPage | Vecteurs pgvector, chunks, knowledge base

== User Stories

=== US-DOC-01 — En tant que redacteur technique, je veux generer un squelette AsciiDoc via IA

*Priorite* : P1
*EPIC* : DOC-2

.Valeur
Gagner du temps sur la redaction de documentation technique : un prompt
structure produit un document AsciiDoc coherent, que l'humain valide et enrichit.

.Criteres d'acceptation
- La tache `generateDocument` accepte un prompt (fichier ou inline)
- Le LLM (deepseek-v4-pro via langchain4j) produit un document AsciiDoc structure
- Le document respecte les conventions AsciiDoc du workspace (sectionnement, attributs, include)
- Un fichier `.adoc` valide est ecrit dans le repertoire de sortie
- La source n'est jamais modifiee — seul le fichier genere est produit

=== US-DOC-02 — En tant que formateur, je veux convertir un SPG AsciiDoc en PDF stagiaire

*Priorite* : P1
*EPIC* : DOC-4

.Valeur
Un SPG/SPD redige en AsciiDoc devient un PDF imprimable pour le stagiaire,
avec mise en page professionnelle (Asciidoctor PDF).

.Criteres d'acceptation
- `convertDocumentToPdf` produit un PDF depuis un fichier `.adoc` source
- Le theme PDF est configurable (attributs Asciidoctor, fichier theme YML)
- Les tableaux, listes, blocs de code et admonitions sont rendus correctement
- Le PDF est consultable sur mobile et desktop

=== US-DOC-03 — En tant que developpeur, je veux convertir une spec AsciiDoc en HTML5

*Priorite* : P1
*EPIC* : DOC-3

.Valeur
Une documentation technique en AsciiDoc devient une page HTML navigable,
ideale pour publication sur site statique ou GitHub Pages.

.Criteres d'acceptation
- `convertDocumentToHtml` produit un fichier HTML5 valide
- Le rendu inclut la table des matieres, les ancres, la syntaxe colorisee
- Les includes AsciiDoc sont resolus correctement
- Le fichier HTML s'affiche correctement dans Chrome, Firefox, Safari

=== US-DOC-04 — En tant que lecteur mobile, je veux exporter une formation en EPUB

*Priorite* : P2
*EPIC* : DOC-5

.Valeur
Les sessions de formation sont lisibles hors-ligne sur liseuse Kindle/Kobo
ou application de lecture EPUB.

.Criteres d'acceptation
- `convertDocumentToEpub` produit un fichier `.epub` valide
- La structure du document (chapitres, sections) est preservee dans la navigation
- Les images sont integrees au format EPUB
- Le fichier passe la validation epubcheck

=== US-DOC-05 — En tant qu'integrateur, je veux que les documents soient consommables par runner-gradle N3

*Priorite* : P2
*EPIC* : DOC-6

.Valeur
Les documents generes et convertis sont indexes dans le composite-context
du runner-gradle pour deploiement automatique.

.Criteres d'acceptation
- `collectDocumentRetrieve` produit un `metadata.json` conforme au contrat N3
- Le JSON contient : chemins des fichiers, formats, metadonnees de la source AsciiDoc
- `runner-gradle assembleCompositeContext` ingere le JSON sans erreur

=== US-DOC-06 — En tant que redacteur technique, je veux generer de la documentation technique au format DocBook

*Priorite* : P3
*EPIC* : DOC-5

.Valeur
La documentation technique des plugins Gradle peut etre publiee au format DocBook 5,
standard pour la documentation d'API et les manuels de reference.

.Criteres d'acceptation
- `convertDocumentToDocBook` produit un fichier DocBook 5 valide
- Les elements DocBook (sections, para, programlisting) sont correctement generes
- Le fichier XML DocBook passe la validation RelaxNG

=== US-DOC-07 — En tant que developpeur, je veux une page de manuel UNIX depuis mon AsciiDoc

*Priorite* : P3
*EPIC* : DOC-5

.Valeur
Une tache Gradle produit une page de manuel (format troff) directement
depuis le document AsciiDoc source, pour integration dans le systeme.

.Criteres d'acceptation
- `convertDocumentToManPage` produit un fichier de page de manuel valide
- La section (1-8) est configurable via l'extension
- `man <fichier>` affiche correctement le contenu

== Notes

* AsciidoctorJ est une dependance directe (implementation) — pas via codex-gradle.
  Les deux boroughs partagent la version via workspace-bom.
* La generation IA peut etre desactivee (fallback template) si le LLM n'est pas disponible.
* Boundary claire : Codex = READ + RAG (ingestion, vectorisation, recherche semantique).
  Document = WRITE + PUBLISH (creation IA, conversion multi-format).
  Voir `.agents/ARCHITECTURE_BOUNDARY.adoc`.
* Le LLM bridge est partage avec planner-gradle en compileOnly pour eviter duplication.
* Priorite MVP0 : PDF stagiaire et HTML site statique > EPUB > DocBook > ManPage.

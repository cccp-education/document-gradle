== Vision Produit

Un plugin Gradle N2 qui **cree du contenu AsciiDoc assiste par IA**
et le **publie dans tous les formats supportes par AsciidoctorJ**
(HTML5, PDF, EPUB3, DocBook 5, ManPage), sans sortir du pipeline Gradle.

Le plugin couvre deux missions complementaires :

1. **Generation IA** → un LLM (gpt-oss:120b-cloud) assiste la redaction
   de documentation, specs, manuels, rapports en AsciiDoc structure
2. **Publication multi-format** → chaque backend AsciidoctorJ est expose
   comme une tache Gradle isolee avec configuration fine

'''
== Boundary Codex (Brooklyn)

_Document = WRITE + PUBLISH. Codex = READ + RAG._

Voir `.agents/ARCHITECTURE_BOUNDARY.adoc` pour le detail complet.

[cols="2,5,5"]
|===
| | Document (ici) | Codex (Brooklyn)
| Mission | Creer + publier | Ingerer + chercher
| Entree | Prompt LLM ou AsciiDoc source | PDF, EPUB, AsciiDoc existant
| Sortie | HTML, PDF, EPUB, DocBook, ManPage | Vecteurs pgvector, chunks, knowledge base
| AsciidoctorJ | Implementation directe | Parsing AST + export KB
| LLM | Oui (langchain4j) | Non
| Base de donnees | Aucune | PostgreSQL + pgvector
|===

=== US-DOC-09 — En tant qu'editeur, je veux personnaliser le theme visuel de mon document

*Priorite* : P1
*EPIC* : DOC-10
*Statut* : ✅ TERMINÉ (session 008)

.Valeur
Le document final (PDF, HTML, EPUB) respecte la charte graphique du projet
(couleurs, polices, logos, en-tete/pied de page) via un systeme de theming
configurable dans le DSL Gradle.

.Decision architecture (session 008) — Theme = value object, pas task séparée
- **Option retenue** : `DocumentTheme` value object injecté à la conversion par attributs AsciidoctorJ
- Le thème n'est pas une tâche `applyTheme` distincte — il est câblé dans `ConvertDocumentTask`
- `toAttributes(format)` mappe vers les attributs AsciidoctorJ par backend :
  * HTML → `stylesheet` (CSS inline dans `<style>`)
  * PDF → `pdf-theme` + `pdf-front-cover-image` (logo)
  * EPUB → `epub-stylesheet`
  * DocBook/ManPage → vide (non supporté)
- DSL nested `document { theme { } }` via `fun interface Action` (SAM)
- Fallback : `DocumentTheme()` vide = défauts AsciidoctorJ

.Criteres d'acceptation
- ✅ Le DSL `document { }` expose une section `theme { }` avec :
  * `pdfTheme.set(file("talaria-theme.yml"))` — fichier YML Asciidoctor PDF
  * `htmlStylesheet.set(file("talaria.css"))` — feuille de style CSS
  * `epubStylesheet.set(file("epub.css"))` — CSS embarque dans l'EPUB
  * `logo.set(file("logo.png"))` — image titre pour PDF
- ✅ Le theme YML PDF supporte : couleurs, polices, marges, en-tete/pied de page (via Asciidoctor PDF)
- ✅ Le theme CSS HTML supporte : polices, couleurs, code highlighting, responsive (via AsciidoctorJ)
- ✅ Fallback : theme par defaut integre si aucun fichier fourni (DocumentTheme vide)

== Notes

* AsciidoctorJ est une dependance directe (implementation) — pas via codex-gradle.
  Les deux boroughs partagent la version via workspace-bom.
* La generation IA peut etre desactivee (fallback template) si le LLM n'est pas disponible.
* Boundary claire : Codex = READ + RAG (ingestion, vectorisation, recherche semantique).
  Document = WRITE + PUBLISH (creation IA, enrichissement, conversion multi-format).
  Voir `.agents/ARCHITECTURE_BOUNDARY.adoc`.
* Le LLM bridge est partage avec planner-gradle en compileOnly pour eviter duplication.
* plantuml-gradle est en compileOnly (composition) — un diagramme est un constituant
  atomique du document, pas un pair N2. Relation contenant→contenu.
* Priorite MVP0 : PDF stagiaire et HTML site statique > EPUB > DocBook > ManPage.
* Pipeline livre (photos → OCR → AsciiDoc → livre multi-format) : voir US-DOC-11.
  Codex gere l'OCR, Document gere la publication. Orchestration par runner-gradle N3.
* Le theme YML PDF et le CSS HTML/EPUB sont des fichiers versionnes dans le projet,
  pas generes par le plugin. Document les applique, ne les cree pas.

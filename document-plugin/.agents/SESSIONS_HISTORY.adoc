== Session 010 (2026-07-03) — DOC-6 N3 runner-gradle integration

**DOC-6** : Implémentation de `collectDocumentRetrieve` — la tâche Gradle qui produit
les fichiers JSON consommables par runner-gradle N3 (`assembleCompositeContext`).

**Ubiquitous language (DDD)** :
- `DocumentMetadata` — provenance (source="new-orleans", type, sessions, generatedAt, model, version, dependencies)
- `DocumentArtifactEntry` — un artefact produit (path, format, sourceAdoc, exists) + toMap
- `DocumentRetrieveResult` — enveloppe N3 (source, entries, count, timestamp) + writeTo
- `DocumentArtifactCollector` — scan outputDir → entries (logique testable sans Gradle)
- `CollectDocumentRetrieveTask` — task Gradle (orchestre collector + result + metadata)

**Contrat N3 de facto** (pas de contrat N0 formel) :
- `composite-context.json` : `{source, entries, count, timestamp}` — runner lit `entries`
- `metadata.json` : `{source, type, sessions, generatedAt, model, version, dependencies}`

**Méthode baby-step TDD/BDD** : 8 baby-steps Red→Green, pyramide unit→functional→cucumber.

**Nouveaux fichiers** : 5 main + 6 test (DocumentMetadata, DocumentArtifactEntry, DocumentRetrieveResult, DocumentArtifactCollector, CollectDocumentRetrieveTask + tests associés)

**Fichiers modifiés** : DocumentPlugin.kt (wiring remplace stub), DocumentPluginFunctionalTest.kt (+2), document.feature (+2 scénarios @doc6), DocumentSteps.kt (+1 Given +5 Then), DocumentWorld.kt (+1 factory +2 helpers)

**Tests** : 135 verts (90 unit + 23 functional + 22 cucumber), 0 failures (+32 vs session 008)

**Loi de l'Économie d'Encre** : `collectDocumentRetrieve` est READ-only — indexe les artefacts produits, n'invoque jamais le LLM.

== Session 012 (2026-07-04) — DOC-11 Book Pipeline Codex -> Document

**DOC-11** : Pipeline livre complet côté Document — assembler les pages
AsciiDoc OCRisées (produites par codex-gradle) en un livre unique,
incruster les photos originales, et exposer une tâche composite
`bookPipeline` reproductible en une commande.

**Boundary respectée** : l'OCR est dans Codex (Brooklyn). Document (New
Orleans) lit l'AsciiDoc sur disque (pont N2 <-> N2), jamais de dépendance
directe vers codex-gradle.

**Ubiquitous language (DDD)** :
- `BookPage` — une page OCRisée (file, order, photo optionnelle)
- `PageOrder` — value class inline, ordre dérivé du préfixe numérique
- `BookAssemblyResult` — résultat d'assemblage (content, pages, photoCount)
- `BookAssembler` — assembleur pages -> livre AsciiDoc (logique testable sans Gradle)
- `AssembleBookTask` — task Gradle (orchestre BookAssembler + writeTo)

**Méthode baby-step TDD/BDD** : 9 baby-steps, pyramide unit -> functional -> cucumber.
- BDD : 5 scénarios `@doc11 @book` en anglais (assembleBook, bookPipeline, assemblage, photos, pipeline complet)
- DDD : 3 value objects (BookPage, PageOrder, BookAssemblyResult)
- TDD : 11 tests unitaires BookAssembler (merge, ordre, photos, immutabilité, filtre)
- Functional : 3 tests (assembleBook merge, assembleBook photos, bookPipeline registration)
- Cucumber : 5 scénarios end-to-end

**Nouveaux fichiers** : 5 main + 1 test (BookPage, PageOrder, BookAssemblyResult, BookAssembler, AssembleBookTask, BookAssemblerTest)

**Fichiers modifiés** : DocumentExtension.kt (+4 propriétés book), DocumentPlugin.kt (registerAssembleBook + registerBookPipeline + conventions), document.feature (5 scénarios DOC-11 + traduction intégrale anglais), DocumentSteps.kt (+3 Given +4 Then), DocumentWorld.kt (+1 factory +1 helper), DocumentPluginFunctionalTest.kt (+3 tests)

**Tests** : 155 verts (100 unit + 27 functional + 28 cucumber), 0 failures (+18 vs session 011)

**Décisions techniques** :
- PageOrder par préfixe numérique (`001-page.adoc` -> 1), non-numérotées après
- Photos optionnelles (png/jpg/jpeg/webp, même baseName que la page)
- bookPipeline = lifecycle task avec `mustRunAfter` pour l'ordre
- Gherkin traduit intégralement en anglais (règle code en anglais)

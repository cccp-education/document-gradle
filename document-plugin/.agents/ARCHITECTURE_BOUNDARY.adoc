= ARCHITECTURE_BOUNDARY — Document Gradle (New Orleans 🟢) ↔ Codex Gradle (Brooklyn) + plantuml-gradle (HTOWN)
:toc:
:date: 2026-06-14
:status: [STRATEGIE] — decision validee, implementation en attente

'''
== Problème Identifié

L'architecture initiale (session 000) documentait une dépendance
`compileOnly` de document-gradle vers codex-gradle pour AsciidoctorJ.
Ce couplage posait trois problèmes :

. **Chevauchement fonctionnel** : codex a déjà `ExportKnowledgeBaseTask`
  qui produit JSON-LD/Markdown/AsciiDoc. La conversion HTML/PDF/EPUB est
  le prolongement naturel de cette pipeline, ce qui crée une boundary floue.
. **Couplage fragile** : document dépend de codex pour son moteur de conversion
  principal (AsciidoctorJ). Si codex change de version, document casse.
  AsciidoctorJ n'est pas la propriété de codex.
. **Sémantique floue** : "codex" (livre manuscrit) vs "document"
  (document numérique) — les deux manipulent AsciiDoc, les deux parlent
  de documents. La frontière "codex = pipeline lourd, document = conversion
  légère" est arbitraire et non maintenable.

'''
== Boundary Claire — Codex READ, Document WRITE

.Les deux boroughs sont parallèles (N2), pas empilés
[cols="2,5,5"]
|===
| Aspect | Codex (Brooklyn) — READ + RAG | Document (New Orleans) — WRITE + PUBLISH

| Mission
| Ingérer des documents existants, en extraire la structure,
  les vectoriser, les rendre cherchables sémantiquement.
| Générer du contenu AsciiDoc assisté par IA, puis le convertir
  dans tous les formats de publication (HTML, PDF, EPUB, etc.).

| Verbe DAG
| COLLECT → TRANSFORM → DEPLOY (knowledge base RAG)
| GENERATE → TRANSFORM → DEPLOY (publication multi-format)

| Entrée
| PDF, EPUB, AsciiDoc *existants* (provenant de `books-collection/`
  ou du pipeline de formation) + *photos de pages* (OCR via LLM)
| Prompt LLM → AsciiDoc *généré* OU AsciiDoc *source* écrit par l'humain
  OU AsciiDoc *OCRisé* produit par codex (fichier disque, pont N2↔N2)

| Sortie
| Vecteurs pgvector, chunks sémantiques, knowledge base (JSON-LD),
  composite-context.json pour runner-gradle (ContextChannel.Docs)
| HTML5, PDF, EPUB3, DocBook 5, ManPage, metadata.json

| AsciidoctorJ
| Utilisé pour le *parsing AST* (AsciiDoc → LddNode)
  et l'export knowledge base en AsciiDoc
| Utilisé pour la *conversion backend* (AsciiDoc → HTML/PDF/EPUB/...).
  *Dépendance directe*, pas via codex.

| LLM
| Oui — gpt-oss:120b-cloud pour OCR photos → AsciiDoc structuré
  (tâche `collectOcr`, contrats `codex.ocr.OcrRequest/Result/Config`)
| Oui — gpt-oss:120b-cloud via langchain4j pour
  generation de contenu AsciiDoc (tâche `generateDocument`)

| Vectorisation
| Oui — ONNX AllMiniLmL6V2 + pgvector
| Non

| Base de données
| PostgreSQL avec pgvector (R2DBC)
| Aucune
|===

'''
== DAG — Placement

----
N0 → graphify-gradle (knowledge graph)
       │
N1 → codebase-gradle (contrats partagés ContextChannel/CompositeContext)
       │
N2 → ┌─ codex-gradle (READ + RAG)
     │    extract PDF/EPUB → chunk → vectorize → semantic search
     │    + OCR photos → AsciiDoc structuré (collectOcr)
     │
     ├─ plantuml-gradle (HTOWN) — génération diagrammes
     │    prompt LLM → PlantUML → PNG/SVG
     │    *Composition* : constituant atomique du document
     │
     └─ document-gradle (WRITE + PUBLISH)
          prompt LLM → generate AsciiDoc → enrich (PlantUML/images/HTML) → convert HTML/PDF/EPUB/DocBook/ManPage
       │
N3 → runner-gradle (assembleCompositeContext, orchestration, deploy gh-pages)
----

Codex et document sont *parallèles* en N2. Aucun lien de dépendance direct.
Les deux exposent leur output via metadata.json que runner-gradle (N3) assemble.

plantuml-gradle est en *composition* avec document-gradle (contenant→contenu).
Un diagramme est un constituant atomique du document, pas un pair N2.
Dépendance compileOnly légitime, pas de cycle DAG.

'''
== Boundary PlantUML (HTOWN) — Composition, pas Parallelisme

*plantuml-gradle est un composant du document, pas un pair.*

Contrairement a la relation Codex↔Document (parallele N2, missions orthogonales),
la relation Document→PlantUML est une relation de *composition* :
un diagramme est un element constitutif du document, au meme titre qu'une
image ou un bloc de code. C'est la meme relation qu'un `<img>` dans du HTML.

[cols="2,5,5", options="header"]
|===
| Aspect | Document (New Orleans) — Contenant | plantuml-gradle (HTOWN) — Contenu
| Relation | Contenant — incruste le PNG dans le document final | Contenu — genere le PNG sur disque
| Dependance | compileOnly (contrat de generation) | Aucune vers Document
| Flux | `enrichDocument` → `plantumlGenerate` → `image::diagram.png[]` | Prompt LLM → PlantUML → PNG/SVG
| Alternative | Extension Asciidoctor Diagram (inline, sans fichier intermediaire) | —
| Ordre N3 | runner-gradle execute plantumlGenerate avant enrichDocument | —
|===

*Regle* : plantuml-gradle est deja publie sur Maven Central (0.0.1), stable (380/380 PASS).
Document peut le referencer en compileOnly sans risque de cycle DAG.
La seule contrainte est l'ordre d'execution dans runner-gradle N3,
ce qui est le job de l'orchestrateur, pas une raison d'interdire la dependance.

'''
== Pipeline Livre — Codex OCR → Document Publication

Cas d'usage complet : produire un livre multi-format (HTML/PDF/EPUB)
a partir de photos de pages d'un ouvrage existant.

----
[CODEX N2] collectOcr ── photo PNG/JPEG → LLM gpt-oss:120b-cloud → OcrResult.structuredText (AsciiDoc)
                │
                ├─▶ transformChunk → collectIngest → pgvector (RAG)
                │
                └─▶ AsciiDoc source (fichier disque, pont N2↔N2)
                        │
[DOCUMENT N2] ─────────┘
    ├─ generateDocument (optionnel : regenerer/remixer via LLM)
    ├─ enrichDocument
    │    ├─ PlantUML (plantuml-gradle → diagram.png → image::[])
    │    ├─ Images (photos originales en illustration)
    │    ├─ HTML passthrough (++++...++++)
    │    └─ Includes (include::chapitre.adoc[])
    ├─ applyTheme (theme.yml PDF, talaria.css HTML, epub.css)
    ├─ convertDocumentToHtml → HTML5 navigable (TOC, ancres)
    ├─ convertDocumentToPdf  → PDF imprimeur (theme pro, pagination)
    └─ convertDocumentToEpub → EPUB3 valide (epubcheck)
            │
[RUNNER N3] assembleCompositeContext → deploy gh-pages
----

*Contrats OCR* : `codex.ocr.OcrRequest` (image bytes + format + langue + prompt),
`codex.ocr.OcrResult` (structuredText AsciiDoc + confidence + metadata),
`codex.ocr.OcrConfig` (provider/model/maxTokens/temperature).
Implementes dans codex-gradle (session 027), tests en cours (CDX-15, session 030).

*Pont N2↔N2* : l'AsciiDoc sur disque. Codex ecrit, Document lit.
Aucune dependance Gradle directe entre les deux plugins.

'''
== Contrat Inter-Borough

=== Ce qui est partagé (N0 via workspace-bom)

- `codebase-contracts` : `ContextChannel`, `ChannelBudget`, `CompositeContext`
- `llm-pool-contracts` : pool de clés API (pour la partie LLM de document)

=== Ce qui est privé (pas de dépendance directe)

- *AsciidoctorJ* : chaque plugin a sa propre dépendance. Aucun `compileOnly` croise.
- *Langchain4j* : propre a document (pour la generation IA).
  Codex ne fait pas de LLM.
- *pgvector* : propre a codex. Document ne fait pas de RAG.

=== Ce que chaque borough expose a runner-gradle (N3)

[cols="3,5"]
|===
| Borough | Contrat N3
| Codex
| `composite-context.json` — résultats de recherche sémantique
  (ContextChannel.Docs), chunks avec similarite cosine,
  source documents, metadata provenance

| Document
| `metadata.json` — chemins des fichiers generes, formats de sortie,
  metadonnees de la source AsciiDoc, provenance IA (modele, prompt, date)
|===

'''
== Règle d'Or

----
CODEX lit les documents existants et les rend cherchables.
  → OCR photos, extraction PDF/EPUB, vectorisation, RAG.
DOCUMENT crée, enrichit et publie des documents dans tous les formats.
  → Generation IA, incrustation PlantUML/images/HTML, theming,
    conversion HTML/PDF/EPUB/DocBook/ManPage.

Quand un AsciiDoc est produit (par planner, codex OCR, ou humain),
il est source pour les DEUX boroughs : codex le vectorise pour le RAG,
document le convertit en livrable multi-format.

PLANTUML est un constituant du document, pas un pair.
  → plantuml-gradle genere le diagramme, document l'incruste.
  → Relation contenant→contenu, dependance compileOnly legitime.
----

== Risques Residuels

[cols="1,3,2,2", options="header"]
|===
| ID | Risque | Impact | Mitigation
| BND-01
| Codex duplique une tache de conversion (ex: ExportKnowledgeBase
  fait deja de l'AsciiDoc→Markdown)
| Moyen
| Codex garde l'export *knowledge base* (JSON-LD structure) mais les
  conversions de *publication* (HTML/PDF/EPUB) restent dans document.
  ExportKnowledgeBase ne produit pas de HTML/PDF/EPUB livrable stagiaire.

| BND-02
| Deux versions d'AsciidoctorJ cohabitent (codex vs document)
| Faible
| Les deux utilisent la meme version via le BOM workspace-bom.
  Pas de version mismatch possible.

| BND-03
| plantuml-gradle non execute avant enrichDocument → diagrammes manquants
| Moyen
| Runner-gradle N3 garantit l'ordre : plantumlGenerate avant enrichDocument.
  Fallback : placeholder texte si PNG absent.

| BND-04
| Cout tokens OCR — 300 pages × gpt-oss:120b-cloud
| Moyen
| OcrConfig.maxTokens controle le budget par page.
  Temperature 0.0 pour resultats deterministes.
  Pool de cles API (api-key-pool-gradle) pour rotation.

| BND-05
| Asciidoctor Diagram necessite Graphviz natif (dot)
| Moyen
| Alternative : plantuml-gradle genere le PNG en amont,
  Document incruste via image::[]. Pas de dependance Graphviz
  si le mode fichier intermediaire est utilise.
|===

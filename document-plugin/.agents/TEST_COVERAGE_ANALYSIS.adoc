== Notes

- DocumentConverter : objet domaine convertisseur (5 formats — html5/pdf/epub3/docbook5/manpage).
- Distinction texte (hash dans fichier, shouldSkipConversion) vs binaire (sidecar .sourcehash, shouldSkipBinaryConversion).
- standalone(true) requis pour doctype HTML/DocBook/ManPage complet.
- asciidoctorj-pdf 2.3.23 : module séparé (backend pdf, non inclus dans core).
- asciidoctorj-epub3 2.2.0 : module séparé (backend epub3, non inclus dans core).
- DocBook5 + ManPage : natifs à asciidoctorj core (aucune extension requise).
- ManPage requiert `:doctype: manpage` + section NAME + SYNOPSIS dans la source.
- Économie d'encre : skip si output existe et hash source correspond (texte ou binaire).
- convertBinary factorise PDF + EPUB (skip par sidecar, convertToFile, writeBinaryMetadataHeader).
- DocumentEnricher (DOC-9) : résolution récursive des include:: (max depth 10), préservation
  native des blocs [plantuml] et ++++ (pas d'échappement — l'enrichisseur ne touche pas
  la syntaxe AsciiDoc, il résout uniquement les includes).
- EnrichDocumentTask : @DisableCachingByDefault + shouldSkipConversion (hash source en metadata).
- Asciidoctor Diagram inline : rendu des blocs [plantuml] à la conversion par le bundle déjà présent.
- DocumentTheme (DOC-10) : value object immutable (pdfTheme, htmlStylesheet, epubStylesheet, logo).
  toAttributes(format) mappe vers les attributs AsciidoctorJ par backend (stylesheet, pdf-theme,
  epub-stylesheet, pdf-front-cover-image). DocBook/ManPage non supportés (map vide).
- AsciidoctorJ inline le CSS HTML dans <style> (pas de <link> ni mot "stylesheet" dans le HTML).
  Le test BDD vérifie `font-family` (contenu du CSS) plutôt que le mot "stylesheet".
- buildOptions centralise la construction des Options + injection Attributes par backend.
- DSL nested theme { } via fun interface Action (SAM) — pattern planner/codebase respecté.
- Pattern surcharge : CLI (-Pdocument.pdfTheme) > DSL (theme { }) > convention (vide).
- @PathSensitive(NONE) + @Optional sur les 4 propriétés thème (validation Gradle OK).
- DocumentMetadata (DOC-6) : provenance N3 conforme au contrat de facto (source, type, sessions,
  generatedAt, model, version, dependencies). forNewOrleans() factory + writeTo() JSON.
- DocumentArtifactEntry (DOC-6) : entry N3 (path, format, sourceAdoc, exists) + toMap() pour
  la sérialisation dans l'array `entries` du composite-context.json.
- DocumentRetrieveResult (DOC-6) : enveloppe N3 (source="new-orleans", entries, count, timestamp)
  + writeTo() produit composite-context.json. Runner-gradle lit uniquement la clé `entries`.
- DocumentArtifactCollector (DOC-6) : scan outputDir → list of DocumentArtifactEntry. Logique
  extraite de la task pour être unit-testable sans Gradle runtime. Ne retourne que les fichiers
  existants (pas de faux positives).
- CollectDocumentRetrieveTask (DOC-6) : @DisableCachingByDefault, orchestre collector + result
  + metadata. READ-only — n'invoque jamais le LLM (Loi de l'Économie d'Encre).
- Contrat N3 de facto : pas de contrat N0 formel (workspace-bom pipeline-contracts MEM-2 non
  implémenté). Chaque borough duplique une classe Metadata. Document suit le pattern hyperframes.

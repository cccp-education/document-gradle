== Objectif

Pipeline complet : photos -> OCR Codex -> AsciiDoc -> enrichissement ->
publication multi-format. Côté Document : assembler les pages OCR-isées en
un livre unique, incruster les photos, appliquer le thème, convertir en
HTML/PDF/EPUB, et exposer une tâche composite `bookPipeline` reproductible
en une commande.

Boundary respectée : l'OCR est dans Codex (Brooklyn). Document (New Orleans)
lit l'AsciiDoc sur disque (pont N2 <-> N2), jamais de dépendance directe
vers codex-gradle.

== Décisions techniques

. *Pas d'OCR côté Document* : boundary respectée, Codex produit l'AsciiDoc
  sur disque, Document le consomme (Rule 7, lecture seule).
. *Pont N2 <-> N2 = filesystem* : aucun lien Gradle direct entre codex et
  document. `bookPagesDir` pointe vers l'output de Codex `collectOcr`.
. *PageOrder par préfixe numérique* : `001-page.adoc` -> 1, `appendix.adoc`
  -> Int.MAX_VALUE (après les numérotées). Déterministe, alphabétique
  comme tiebreaker.
. *Photos optionnelles* : `BookPage.fromFile` cherche png/jpg/jpeg/webp
  dans `photosDir` avec le même baseName que la page.
. *bookPipeline = lifecycle task* : `dependsOn` + `mustRunAfter` pour
  l'ordre sans court-circuiter l'up-to-date check des sous-tasks.
. *Gherkin en anglais* : traduction intégrale du `.feature` existant
  (règle "code en anglais" appliquée rétroactivement).

== Risques résiduels

. *R-06 (coût tokens OCR)* : non adressé côté Document — c'est le job de
  Codex (OcrConfig.maxTokens, pool de clés).
. *BND-03 (plantuml-gradle non exécuté avant enrichDocument)* : bookPipeline
  n'invoque pas plantumlGenerate. Runner-gradle N3 reste responsable de
  l'orchestration cross-borough.
. *bookPipeline ordre photos* : `mustRunAfter` garantit l'ordre seulement
  si toutes les tasks sont dans le graph. Si l'utilisateur invoque
  `convertDocumentToHtml` seul, assembleBook ne tourne pas. C'est le
  comportement attendu (task isolée).

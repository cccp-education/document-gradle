== DSL unifié — exemple

```kotlin
document {
    source.set(file("livre.adoc"))
    enrich {
        plantuml.set(true)
        images.set(true)
        passthrough.set(true)
    }
    outputs {
        html.set(true)
        pdf.set(true)
        epub.set(true)
    }
    theme {
        pdfTheme.set(file("theme.yml"))
        htmlStylesheet.set(file("style.css"))
    }
    metadata {
        title.set("Mon Livre")
        author.set("Auteur")
        language.set("fr")
    }
}
```

== Contrat N3 — document-config.json

```json
{
  "source"  : "livre.adoc",
  "enrich"  : { "plantuml": true, "images": true, "passthrough": true },
  "outputs" : { "html": true, "pdf": true, "epub": true, "docbook": false, "manpage": false },
  "theme"   : { "pdfTheme": "...", "htmlStylesheet": "...", "epubStylesheet": "...", "logo": "..." },
  "metadata": { "title": "Mon Livre", "author": "Auteur", "language": "fr" }
}
```

== Décision technique — Nested DSL via `org.gradle.api.Action<T>`

Les nested blocks `enrich { }`, `outputs { }`, `metadata { }` utilisent
`fun enrich(action: Action<DocumentEnrichDsl>)` (standard Gradle) plutôt que
des `fun interface` SAM — c'est la signature que le Gradle Kotlin DSL attend
pour générer les lambdas à receiver. Les classes DSL sont concrètes avec des
`val` initialisés via `project.objects.property()` dans le plugin, pour
permettre la résolution des propriétés (`plantuml.set(true)`) dans le block.

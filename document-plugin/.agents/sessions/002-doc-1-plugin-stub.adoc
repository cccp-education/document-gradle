== Méthode

2. **TDD** — Red → Green → Refactor, 1 test à la fois

== Pattern de surcharge

Pattern or planner/codebase (le plus élégant du workspace) :
```
CLI (-Pdocument.xxx) > DSL (block document { }) > convention (.convention())
```
Via `project.providers.gradleProperty("document.xxx").orElse(ext.xxx)` — lazy, déclaratif, zéro ConfigMerger.

Étude comparative menée sur 14 boroughs :
- Pattern A (legacy manual merger) : bakery, plantuml, capsule — 300-400 lignes ConfigMerger
- Pattern B (modern lazy providers) : planner, codebase, codex — zéro ConfigMerger, cascade déclarative
- Pattern C (var mutable / vestige) : container-provision, slider, notebook — anti-pattern

Choix : Pattern B (planner/codebase) — élimine ~700 lignes de ConfigMerger par borough.

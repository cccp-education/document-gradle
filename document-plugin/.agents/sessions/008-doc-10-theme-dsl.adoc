== Contexte

Session 008 — implémentation de l'EPIC DOC-10 (`theme { }` DSL). Le document
AsciiDoc converti peut recevoir un thème visuel personnalisé : feuille de
style CSS pour HTML, thème YML pour PDF, CSS pour EPUB, et image logo pour
la couverture PDF. Le thème est configurable via un block DSL imbriqué
`document { theme { } }` et injecté au moment de la conversion par attributs
AsciidoctorJ (`:stylesheet:`, `:pdf-theme:`, `:epub-stylesheet:`,
`:pdf-front-cover-image:`). Fallback : thème vide = défauts AsciidoctorJ.

Méthode baby-step DDD/TDD/BDD respectée : BDD Cucumber en amont (Red),
TDD unit Red → Green, functional GradleTestKit, refactor.

== Décisions techniques

. **Theme = value object, pas task séparée** — le thème est appliqué au
  moment de la conversion (attributs AsciidoctorJ injectés dans les Options),
  pas par une tâche `applyTheme` distincte. Moins de tâches, moins de I/O.
. **AsciidoctorJ inline le CSS HTML** — l'attribut `stylesheet` fait qu'AsciidoctorJ
  lit le fichier CSS et l'incorpore dans une balise `<style>` (pas de `<link>`.
  Le test vérifie donc `font-family` (contenu du CSS) plutôt que le mot
  "stylesheet" (absent du HTML produit).
. **toAttributes par format** — chaque backend a ses propres attributs.
  `stylesheet` pour HTML, `pdf-theme` + `pdf-front-cover-image` pour PDF,
  `epub-stylesheet` pour EPUB. DocBook/ManPage ignorés (non supporté).
. **@PathSensitive(NONE) + @Optional** — les 4 propriétés thème sont
  optionnelles (fallback vide) et normalisées (validation Gradle OK).
. **Pattern surcharge** — `CLI > DSL > convention` respecté via `cliFile()`.
. **Économie d'Encre** — le thème est un `@Input` de la task, donc le hash
  de build Gradle inclut le thème. Si le thème change, la task re-exécute.
  Si la source + thème sont identiques, `shouldSkipConversion` s'applique.
. **Code en anglais** — DocumentTheme, toAttributes, isEmpty, buildOptions.
  Commentaires KDoc en anglais. Aucun français dans le code source.
